# Design: Ускорение Nexus Looker для геораспределённых клиентов amoCRM

Generated by /office-hours on 2026-06-12
Branch: master

Mode: Startup

## Problem Statement

Клиенты amoCRM находятся где угодно — Лондон, Москва, Бали, Нью-Йорк. Вся серверная часть Nexus Looker — один Hetzner-сервер в Хельсинки (`nexus-oko.naithon.one`). Цель: минимальное время от клика по глазику до отрисованного предпросмотра для дальних клиентов. Материковый Китай — вне скоупа (решение владельца).

## Demand Evidence

Честная оговорка: продукт пре-лонч, реальных пользователей ноль. Требование исходит от владельца как подготовка к маркетплейсу — виджет ставится в кабинеты по всему миру, и географию клиентов мы не контролируем. Это forward-looking требование, не подтверждённое болью живых пользователей, поэтому весь план построен на гейтах по замерам: дорогие шаги не выполняются без данных.

## Status Quo — где сейчас теряется время

Поток по форматам (граница приватности — см. CLAUDE.md):

| Формат | Путь | Зависимость от нашего сервера |
|---|---|---|
| pdf, image, svg, text, md | браузер-only (blob) | нет — скорость = S3 amoCRM, вне контроля |
| docx, pptx, xlsx, csv | браузер качает из amo → POST `/preview-host` → MS viewer качает с нас | да: аплоад + публикация |
| doc, xls, ppt, rtf, odt, ods, odp | браузер качает → POST `/convert` → LibreOffice → PDF назад | да: аплоад + конвертация + даунлоад |

Подтверждённые кодом факты (verified 2026-06-12):

1. **nginx отдаёт только HTTP/1.1** — `converter/deploy/nginx-nexus-oko.conf:74`: `listen 443 ssl` без `http2`.
2. **LibreOffice стартует холодным на каждый запрос** — `converter/convert.js:38`: spawn soffice со свежим профилем; холодный старт — секунды CPU.
3. **Preconnect к конвертеру нет** — TLS+TCP рукопожатие (2–3 RTT) начинается в момент POST; из SEA это сотни мс впустую.
4. **Кэша повторных открытий нет** — повторный клик по тому же файлу = полный re-download из amo + re-конвертация.
5. Rate-limit ключуется по `X-Real-IP`, который ставит наш nginx (важно для этапа CDN).

Время полного пути складывается из: фетч из amo S3 (вне контроля) → TLS+TCP к конвертеру → аплоад через океан → [очередь] → конвертация → даунлоад результата (для office ещё хоп MS viewer).

## Constraints

1. **Виджет 0.2.0 на пороге модерации.** Любая клиентская правка = bump версии + пересборка + повторная сверка + модерация. Клиентские изменения НЕ должны блокировать текущую подачу.
2. Требования amoCRM к публичным интеграциям (п. 3.2): никакой догрузки скриптов; build-guard в `build.js` обязан проходить.
3. Приватность: файлы пользователей не оседают нигде дольше необходимого; имена файлов не логируются (инвариант `convert.js` сохранить); RUM-телеметрия в проде-виджете нежелательна.
4. Эшелон лимитов не ослаблять: Origin-auth, `MAX_INFLIGHT`, nginx `limit_req`, `requireAuth` до `express.raw`.
5. Секреты только на сервере; конфиги деплоя — `converter/deploy/`.
6. Материковый Китай вне скоупа.

## Premises (согласованы 2026-06-12)

1. География влияет только на конвертерные форматы; локальные рендеры зависят от S3 amoCRM — вне нашего контроля.
2. Клиентские правки — только в следующую версию виджета; серверные деплоятся сразу и свободно.
3. Пользователей ноль → RUM невозможен; только синтетические замеры; мульти-регион без данных о реальной географии — преждевременная трата.
4. Китай вне скоупа.
5. Холодный старт LibreOffice — главный абсолютный (негеографический) источник задержки `/convert`.
6. http2 — операционная гигиена, не обещанный выигрыш (пересмотрено по Codex-ревью: для «один большой аплоад + один ответ» мультиплексирование h2 почти ничего не даёт; включаем, эффект замеряем).

## Cross-Model Perspective

Codex (gpt-5.5, независимый cold read, 2026-06-12):

- **Steelman:** «низкофрикционный preview-слой, где сервер используется только когда неизбежно; задача — сделать неизбежный серверный путь ощущаемо локальным для глобальных пользователей, не задерживая лонч и не перестраивая инфраструктуру до появления спроса».
- **Ключевая цитата контекста:** *"ZERO production users yet. Any client-side change = re-moderation. Server-side ships freely"* — это доминирует над планом: до модерации высокорычажная работа только серверная.
- **Оспоренная премисса:** №6 (http2 как «confirmed win») — принято, премисса переписана.
- **Принятые правки:** CDN перенесён за гейт замеров (сначала доказать, что сеть — значимая доля времени); добавлены серверные тайминг-логи как замена RUM.
- **Отклонено:** «скип preconnect/кэша» — Codex отвечал на «что сделать за 48 часов»; в нашем горизонте следующая версия виджета всё равно будет, эти пункты бандлятся туда и не отменяются.

## Approaches Considered

### Approach A: Поэтапный с гейтами — ВЫБРАН
Effort: M · Risk: Low. Сервер сразу, CDN и мульти-регион за гейтами замеров, клиент — в следующей версии виджета. Reuses: DI-контракт `createApp({convert})`, существующий деплой, e2e-стенд букмарклета.

### Approach B: Минимум, сервер-only
Effort: S · Risk: Low. Baseline + http2 + warm LibreOffice + логи. Отвергнут: дальние клиенты не получают ничего против сетевой задержки, preconnect/кэш выпадают из плана совсем.

### Approach C: Максимум сразу
Effort: XL · Risk: High. Всё из A без гейтов + мульти-регион (EU/US/SG, geo-DNS) + RUM. Отвергнут: операционка ×3 без единого пользователя; RUM в публичном виджете — риск на модерации; необходимость ничем не доказана.

## Recommended Approach — поэтапный план

### Этап 0 — Baseline (до любых правок) — ~0.5–1 день

- Синтетические замеры из 3 точек: EU (Лондон), US East (Нью-Йорк), SEA (Сингапур — прокси Бали). Дешёвые VPS-часы или внешний сервис — выбрать по факту.
- Матрица: `POST /convert` с фикс-файлами (small ~100КБ `.doc`, medium ~5МБ `.xls`/`.ppt`), `POST /preview-host` (docx, csv). Полный пользовательский путь (с фетчем из amo) — хотя бы из одной точки через букмарклет-стенд (тестовые сделки 3200807/3177663).
- Метрики per-stage (curl `-w`-шаблон): DNS, TCP+TLS, `time_starttransfer − time_pretransfer` (это **аплоад + серверное время вместе** — curl не умеет изолировать конец аплоада), даунлоад, total. По 10+ прогонов на ячейку, фиксируем p50/p95. Полное разложение «сеть vs сервер» становится возможным после этапа 1: `network_ms = total − server_ms (из логов 1.2) − TLS-handshake`.
- Auth проб: пробы шлют поддельный `Origin` кабинета (принятый риск, спека §8); `X-Source-Token` на сторонние VPS не выносить.
- Результат: отчёт-таблица в `work_directory/04_reviews/11_geo_baseline_report.md`.
- **Гейт для этапа 2 (CDN), считается после этапа 1:** для каждого запроса доля `network_ms / total_ms` (формула выше); гейт пройден, если **медиана доли ≥ 30%** на small `.doc` через `/convert` хотя бы в одной из точек SEA/US.

### Этап 1 — Сервер (деплой сразу, модерацию не трогает) — 1–2 дня

**1.1 Прогретый LibreOffice (главный абсолютный выигрыш).**
- unoserver ИЛИ собственный пул long-lived soffice-воркеров — решить при имплементации; критерий выбора: сохранение kill-семантики по таймауту.
- Инварианты: per-job tmp-каталоги; жёсткий таймаут 30с с kill+**restart воркера**; изоляция профилей по воркеру; **размер пула = `CONCURRENCY` (сейчас 2)** — иначе параллельные запросы сериализуются; `MAX_INFLIGHT` и p-limit-семантика сохраняются.
- **Abort-семантика пересмотрена** (находка ревью): на abort клиента воркера НЕ убиваем — job дорабатывает, результат выбрасывается (потолок потерь — 30с CPU). Kill+restart воркера только по таймауту/зависанию. Иначе пользователь, щёлкающий open/close по файлам (или дешёвый абьюз), устраивает restart-шторм и обнуляет тёплую ёмкость пула. Принятый trade-off: брошенный job занимает слот пула до завершения (≤ 30с) — серия abort'ов на тяжёлых файлах временно поднимает queue-wait следующих запросов, но тёплая ёмкость сохраняется.
- Откат: env-флаг (`CONVERT_MODE=warm|cold`) в `server.js` — DI (`createApp({ convert })`) делает подмену реализации тривиальной; старый холодный `convert.js` остаётся как fallback.
- RAM-бюджет: перед деплоем снять фактическую RAM сервера и идл-потребление воркера (~300–500МБ/шт); при нехватке уменьшить пул и/или `MAX_INFLIGHT`.
- Изменения: `converter/Dockerfile` (+unoserver), новый модуль с тем же контрактом `convert(buf, ext, isAborted, target)` — DI-контракт не меняется, тесты auth/лимитов переиспользуются как есть; новые тесты на таймаут/restart и на abort-семантику.
- Acceptance: p50 конвертации small `.doc` < 1.5с; kill по таймауту работает (тест); при пуле N — N параллельных конвертаций без взаимной блокировки; после серии abort'ов следующая конвертация идёт тёплым воркером (без холодного старта), queue-wait ограничен добеганием брошенных джобов (≤ 30с).

**1.2 Серверные тайминг-логи (замена RUM).**
- Лог на запрос: request id, ext, размер, queue wait ms, convert ms, total ms, исход, **грубая гео-метка** — поле «страна» в формате лога. Источник: заголовок `CF-IPCountry` после этапа 2 (в дашборде CF включить IP Geolocation); до этапа 2 — `unknown` (nginx-geoip с лицензией MaxMind ради нуля пользователей не ставим — YAGNI; география синтетических проб и так известна). Приватность: страна — да, IP в лог — нет; имена файлов НЕ логируются (текущий инвариант).
- Request id возвращать клиенту заголовком `X-Request-Id` — джойн синтетических проб с лог-строками при подсчёте гейта (запасной вариант — джойн по таймстампу: сервер тихий, синтетика — единственный трафик).
- Куда: stdout контейнера + docker logging driver с ротацией (~30 дней) — этого ретеншна достаточно для решения по этапу 4.
- Когда появятся реальные пользователи — это готовый источник данных для гейта этапа 4 (деградация по регионам) без телеметрии в клиенте.

**1.3 http2 (гигиена).**
- `listen 443 ssl http2;` в nginx (для nginx ≥ 1.25.1 синтаксис deprecated — использовать отдельную директиву `http2 on;`; проверить версию на сервере). Проверка ALPN: `curl -sI --http2 https://nexus-oko.naithon.one/health`. Эффект не обещаем — попадёт в повторный замер.

**Повторный замер** по матрице этапа 0 → решение по гейту CDN.

### Этап 2 — CDN перед конвертером (только если гейт пройден) — 0.5–1 день

Условие входа: гейт этапа 0 пройден — медиана доли `network_ms/total_ms` ≥ 30% на small `.doc` через `/convert` хотя бы в одной из точек SEA/US (считается после этапа 1).

- Cloudflare-proxy перед `nexus-oko.naithon.one`: edge-TLS (рукопожатие в ближайшем PoP), HTTP/3, аплоад по бэкбону CF. Лимит тела Free-тарифа 100МБ > наших 50МБ — проходит.
- **Обязательные сопутствующие правки** (без них эшелон лимитов ломается/обходится):
  1. Ключ rate-limit `X-Real-IP` → `CF-Connecting-IP`: nginx `set_real_ip_from` CF-диапазоны + `real_ip_header CF-Connecting-IP`; app-код менять не нужно, если nginx продолжает класть итоговый IP в `X-Real-IP` для апстрима — проверить.
  2. Firewall: 443 на origin принимает только IP-диапазоны Cloudflare — иначе лимиты и edge обходятся прямым обращением.
  3. Проверить, что заголовок `Origin` доходит до origin нетронутым (Origin-auth).
  4. `/preview/` остаётся `no-store` (MS качает файл один раз — edge-кэш не нужен и вреден).
  5. **Верифицировать фетчер Microsoft через CF**: `/preview/<uuid>` забирает не-браузерный бот MS — bot-защита Cloudflare (Security Level / Bot Fight Mode) может выдать ему challenge и молча сломать весь office-путь. При необходимости — WAF skip-rule для `/preview/*`.
  6. **Продление TLS origin-сертификата**: с HTTP-01 уходим (за firewall'ом из п.2 прямая валидация ломается, а зависимость продления от CF-прокси — лишняя связка). Предпочтительно **DNS-01**: сертификат остаётся браузер-доверенным, что делает откат CDN тривиальным. CF Origin Certificate (15 лет) — только если осознанно жертвуем простотой отката. Режим SSL в CF — Full (strict).
  7. Ops-хвост: IP-диапазоны CF для `set_real_ip_from` меняются — предусмотреть периодическое обновление списка (скрипт по CF API), иначе лимиты со временем начнут ключеваться по IP PoP'ов (ложные 429 на всех клиентов).
  8. **План отката**: выключить proxy-статус (grey-cloud) в DNS CF + откатить firewall и `real_ip`-конфиг. При DNS-01 (п.6) сертификат остаётся доверенным — откат без outage. Правило: если повторный замер (acceptance ниже) не показал снижения network_ms — откатываемся, CDN «на всякий случай» не держим.
  9. Включить IP Geolocation в дашборде CF — источник `CF-IPCountry` для гео-метки логов 1.2.
- Acceptance: **повторный замер на гейтовой ячейке** (small `.doc`, `/convert`, SEA/US) — network_ms снизился против замера после этапа 1, иначе откат по п.8; TLS-handshake < 100мс из всех 3 точек; эшелон лимитов работает через CF (включая 429); auth-матрица конвертера (200/401/401/415) проходит; **office-путь проходит e2e** (docx через `/preview-host` + реальная загрузка MS-вьювером). Upload-throughput на SEA/US — информационная метрика для сравнения с baseline, порог не ставим.

### Этап 3 — Клиент (в составе СЛЕДУЮЩЕЙ версии виджета) — ~1 день

Не блокирует модерацию 0.2.0; едет в 0.2.1/0.3.0 штатным релиз-циклом (bump `manifest.json` → build → zip).

**3.1 Preconnect.** При открытии модалки для kind ∈ {office, legacy} — прогрев соединения к `nexus-oko.naithon.one`, для office дополнительно к `view.officeapps.live.com`. К моменту POST соединение тёплое — экономия 2–3 RTT (из SEA — до ~секунды).
- Механика (решающие детали): для CORS-POST на конвертер сокет должен быть CORS — `<link rel="preconnect" href="https://nexus-oko.naithon.one" crossorigin>`; без `crossorigin` браузер прогреет non-CORS-сокет и НЕ переиспользует его. Для iframe `view.officeapps.live.com` — наоборот, preconnect **без** `crossorigin`. Вариант с запросом `HEAD /health` — только с `mode: 'no-cors'` (на `/health` нет ACAO-заголовков; иначе CORS-ошибка в консоли кабинета — шум на модерации).
- Ограничение: браузер закрывает неиспользованный прогретый сокет через ~10с. Для файлов, чей фетч из amo дольше, прогрев на открытии модалки протухнет — принимаем: выигрыш в основном для small-файлов (самый частый кейс), стриминг-трекинг прогресса фетча ради повторного прогрева не городим.

**3.2 Сессионный LRU-кэш предпросмотров.** In-memory `Map` href → результат рендера (blob/objectURL или preview-url), только на page load:
- ключ — `href` как есть (без нормализации); кэшируются только успешные рендеры;
- лимиты: по байтам (~100МБ) и по записям (10–20), вытеснение LRU;
- для office: хранить срок жизни preview-url (TTL 5 мин) и не реюзать протухший;
- интеграция с `Loader.dispose()`: кэшированные objectURL не revoke'ить при закрытии модалки, revoke при вытеснении из LRU/unload;
- бонус сверх премиссы 1: кэш покрывает и фетчи из S3 amo (все форматы) — повторное открытие убирает в том числе неподконтрольную нам стадию;
- приватность не меняется: память вкладки, никуда не пишется.
- Тесты: новый `test/cache.test.js` + правки `test/inject.test.js` (поведение модалки покрыто там; при разрастании — выделить modal-тесты в отдельный файл); build-guard проходит.

### Этап 4 — Мульти-регион (по умолчанию НЕ делаем)

Условие входа: есть реальные пользователи И серверные тайминг-логи (1.2, по гео-метке страны) показывают систематическую деградацию из региона, которую CDN не вылечил. Только тогда: реплика конвертера (SG/US, Docker разворачивается легко) + geo-DNS. Нюанс: `/preview-host` от азиатской реплики может не выиграть — файл оттуда забирает Microsoft, чьи фетчеры предположительно в US/EU.

## Вычеркнуто — не делаем

| Что | Почему |
|---|---|
| RUM-телеметрия в проде-виджете | приватность + риск на модерации; заменена серверными тайминг-логами и синтетикой |
| Мульти-регион сейчас | ноль пользователей, ноль данных; за гейтом этапа 4 |
| Китай | решение владельца; GFW не лечится нашей инфраструктурой |
| Компрессия аплоада | docx/xlsx/pptx уже zip-сжаты, выигрыш ~0 |
| Стриминг-рендер PDF | формат требует целиком; iframe-рендер не ускорится |

## Последовательность (итог)

```
Этап 0 (baseline) ──► Этап 1 (warm LO + логи + http2) ──► повторный замер
                                                            │
                       медиана network-доли ≥ 30%          │
                       (small .doc, SEA/US)? ──────────────┼─ да ──► Этап 2 (CDN)
                                                            └─ нет ─► Этап 2 пропускаем

Этап 4 (мульти-регион) ──► только по логам реального трафика; сейчас не планируем
```

## What I noticed about how you think

- Первым вопросом сессии было «файл скачивается на наш сервер или всё происходит в браузере клиента?» — вопрос про границу приватности и архитектуры, а не про фичи. Это вопрос архитектора, и он же определил всю структуру решения.
- «Китай, не учитываем» — скоуп срезан мгновенно и без сожалений, до того как он успел раздуть план.
- «Если что-то реализовывать не нужно, то не делаем» — явная установка на вычёркивание. В спеке из-за неё появилась отдельная таблица «Вычеркнуто» — редкая дисциплина против over-engineering.
