## 1. Проблема, которая была на входе — простыми словами

Большинство форматов (PDF, картинки, текст) виджет показывает прямо в браузере пользователя — для них скорость и так максимальная. Но Office-файлы (.doc, .xls, .docx, таблицы, презентации) браузер сам показать не может — их обрабатывает наш сервер.

Проблема была в том, как сервер это делал: **на каждый клик по глазику он запускал LibreOffice с нуля** — как если бы для открытия каждого документа вы заново включали компьютер, открывали Word, смотрели файл и выключали компьютер. Этот «холодный старт» стоил 1.5–2 секунды даже для крошечного файла. Тяжёлые таблицы вообще подбирались к лимиту времени и иногда обрывались с ошибкой. А повторное открытие того же файла честно проделывало всю работу заново.

Для пользователя из Москвы это означало «подождать пару секунд». Для пользователя с Бали или из Нью-Йорка к этому добавлялась ещё и дорога файла через полпланеты.

## 2. Решение, которое мы применили

Действовали по принципу «сначала измерь, потом строй» — сняли точные замеры **до** любых правок, и они показали: главный тормоз не география, а тот самый холодный старт. Поэтому:

1. **Прогретый пул LibreOffice.** Теперь на сервере постоянно живут заранее запущенные «тёплые» экземпляры LibreOffice — конвертация начинается мгновенно, без запуска с нуля. Пул умеет сам лечиться: зависший или деградировавший экземпляр автоматически перезапускается, ни один пользовательский клик не может «сломать» сервис надолго. На случай проблем есть мгновенный откат к старой схеме одной настройкой.
2. **Умная маршрутизация по размеру.** Маленькие файлы (подавляющее большинство кликов) идут через тёплый пул, очень большие — отдельным путём, чтобы не занимать тёплые экземпляры надолго.
3. **Серверная телеметрия без приватных данных.** Каждый запрос теперь оставляет техническую запись: сколько ждал, сколько конвертировался, из какой страны пришёл. Имена файлов, их содержимое и IP-адреса принципиально не записываются (это отражено в политике конфиденциальности). Эти данные позволят принимать дальнейшие решения по цифрам, а не на глаз.
4. **Подготовлена версия 0.3.0** (не публикуется до решения модерации по 0.2.0): браузер заранее «прогревает» соединение с сервером, пока файл ещё качается, а повторное открытие того же файла происходит мгновенно из кэша вкладки — без повторной загрузки и конвертации.

Каждый шаг прошёл двойное независимое ревью (внутренний агент + codex) и отдельную проверку безопасности; всё покрыто автотестами (90/90 зелёные).

## 3. Результат — замеры с боевого сервера

| Операция | Было | Стало | Изменение |
|---|---|---|---|
| Открытие .doc (типичное КП ~80 КБ) | 1.6 сек | **0.5 сек** | **в 3 раза быстрее** |
| Открытие .csv | 1.5 сек | **0.4 сек** | **в 4 раза быстрее** |
| Тяжёлая таблица 2.8 МБ | ~18 сек, изредка обрыв по таймауту | ~20 сек, **обрывов нет** | надёжнее |
| Повторное открытие того же файла (в 0.3.0) | вся работа заново | **мгновенно из кэша** | — |

Полсекунды на открытие — это уровень «окно появилось сразу». И это ускорение получили **все пользователи разом**, независимо от того, где они находятся.

## 4. Как виджет работает для пользователей в разных частях планеты

Частый вопрос: «а как amoCRM узнает, на какой наш сервер отправить пользователя?» Ответ: **amo вообще не участвует в выборе сервера**. Вот как это устроено.

Виджет — это JS-код, который amoCRM один раз отдаёт в браузер пользователя. Дальше amo в процессе не участвует. Когда пользователь кликает глазик:

- **PDF, картинки, текст, markdown** — браузер берёт файл из хранилища самого amo и показывает его сам. Наши серверы не нужны вообще, где бы они ни стояли.
- **Office-форматы** — браузер пользователя **сам** обращается к нашему домену `nexus-oko.naithon.one`. Это обычный сетевой запрос из браузера, как открытие любого сайта.

Если когда-нибудь понадобятся серверы в нескольких точках планеты, выбор ближайшего происходит автоматически — в момент, когда браузер спрашивает у DNS «какой IP у этого домена?»:

- **GeoDNS**: DNS-провайдер видит, откуда пришёл запрос, и отвечает разными адресами разным людям — москвичу даёт сервер в Хельсинки, пользователю с Бали — сервер в Сингапуре. Код виджета при этом не меняется вообще: домен один, «резолвится» он по-разному.
- **Anycast/CDN** (например, Cloudflare): один и тот же IP-адрес объявлен из сотен точек мира, и интернет-маршрутизация сама доносит трафик до ближайшей.

Наши серверы конвертации полностью автономны (файл пришёл → PDF ушёл, ничего не хранится дольше 5 минут) — региональные копии не нужно синхронизировать между собой.

**Почему мы пока НЕ арендуем серверы по миру:** замеры показали, что главный тормоз был на сервере, а не в сети — и мы его уже убрали. Дорогие шаги (CDN, серверы в Азии/США) поставлены «за гейт»: они будут сделаны только если контрольный замер из дальней точки покажет, что сеть занимает значимую долю времени. Так мы не тратим деньги на инфраструктуру, необходимость которой не доказана, — а телеметрия из п.2 как раз даст эти цифры, когда появятся живые пользователи.
