# Geo-performance: baseline-замеры конвертера (этап 0 спеки 04)

**Дата:** 2026-06-12 · **Задача:** T22 (E4S1) · **Скрипт:** `converter/tools/geo-probe.sh` · **Сырые данные:** `work_directory/tests/fixtures/geo-baseline-local-2026-06-12.csv`

## Методика

- 10 прогонов на ячейку, curl `-w` per-stage тайминги, sleep между прогонами (rate-limit 30/мин на Origin).
- Auth проб: поддельный `Origin: https://geoprobe.amocrm.ru` (принятый риск, спека §8); `X-Source-Token` не использовался и на сторонние машины не выносился.
- **Ограничение curl (спека, этап 0):** `time_starttransfer − time_pretransfer` = аплоад + всё серверное время вместе; для POST с большим телом curl ловит `100 Continue` — starttransfer для conv-medium непоказателен, опираемся на `total`. Раздельное «сеть vs сервер» станет возможным после этапа 1 (T24, серверные логи): `network_ms = total − server_ms − TLS`.

## Выводы baseline

1. **Холодный старт LibreOffice подтверждён численно**: small .doc и крошечный csv — оба ~1.5–1.6с при пренебрежимой сети; это цена per-request spawn (премисса 5 спеки).
2. **conv-medium 18.4с p50 — тревожно близко к таймауту 30с**: тяжёлая xlsx→pdf конвертация 2.8МБ почти исчерпывает бюджет. Warm-пул должен срезать только cold-start долю (~1.5с из 18.4с); остальное — собственно конвертация. Для дальних клиентов к этому добавится аплоад 2.8МБ через океан.
3. **prev-docx 26мс** — публикация без конвертации мгновенна; для office-форматов сервер не является узким местом вообще (узкое место — аплоад клиента и Microsoft viewer).
4. Полный пользовательский путь (клик→рендер через букмарклет-стенд) — будет снят в повторном замере T25 через dev-канал статики (прод-статика 0.2.0 не трогается до решения модерации).
