# Geo-performance: baseline-замеры конвертера (этап 0 спеки 04)

**Дата:** 2026-06-12 · **Задача:** T22 (E4S1) · **Скрипт:** `converter/tools/geo-probe.sh` · **Сырые данные:** `work_directory/tests/fixtures/geo-baseline-local-2026-06-12.csv`

## Методика

- 10 прогонов на ячейку, curl `-w` per-stage тайминги, sleep между прогонами (rate-limit 30/мин на Origin).
- Auth проб: поддельный `Origin: https://geoprobe.amocrm.ru` (принятый риск, спека §8); `X-Source-Token` не использовался и на сторонние машины не выносился.
- **Ограничение curl (спека, этап 0):** `time_starttransfer − time_pretransfer` = аплоад + всё серверное время вместе; для POST с большим телом curl ловит `100 Continue` — starttransfer для conv-medium непоказателен, опираемся на `total`. Раздельное «сеть vs сервер» станет возможным после этапа 1 (T24, серверные логи): `network_ms = total − server_ms − TLS`.

FI-точка при этом идеальна для измерения серверного времени: сеть в ней пренебрежима (health p50 = 29мс), т.е. весь `total` ≈ серверное время.

## Выводы baseline

1. **Холодный старт LibreOffice подтверждён численно**: small .doc и крошечный csv — оба ~1.5–1.6с при пренебрежимой сети; это цена per-request spawn (премисса 5 спеки).
2. **conv-medium 18.4с p50 — тревожно близко к таймауту 30с**: тяжёлая xlsx→pdf конвертация 2.8МБ почти исчерпывает бюджет. Warm-пул должен срезать только cold-start долю (~1.5с из 18.4с); остальное — собственно конвертация. Для дальних клиентов к этому добавится аплоад 2.8МБ через океан.
3. **prev-docx 26мс** — публикация без конвертации мгновенна; для office-форматов сервер не является узким местом вообще (узкое место — аплоад клиента и Microsoft viewer).
4. Полный пользовательский путь (клик→рендер через букмарклет-стенд) — будет снят в повторном замере T25 через dev-канал статики (прод-статика 0.2.0 не трогается до решения модерации).

## Повторный замер (warm, http2) — после деплоя T23–T25 (2026-06-12, точка local/FI)

Деплой: warm-пул unoserver ×2 (`CONVERT_MODE=warm`), тайминг-логи + `X-Request-Id`, nginx http2 (ALPN h2 подтверждён openssl). Сырые данные: `work_directory/tests/fixtures/geo-warm-local-2026-06-12.csv`.

| Ячейка | Cold p50 → **Warm p50** | Cold p95 → **Warm p95** | Δ p50 |
|---|---|---|---|
| health | 0.029с → 0.043с | 0.044с → 0.068с | шум |
| conv-small (.doc 77КБ) | 1.615с → **0.529с** | 1.656с → 1.001с | **−67%** |
| conv-medium (.xlsx 2.8МБ) | 18.407с → 22.963с* | 19.760с → 26.536с | +25%* |
| prev-docx | 0.026с → 0.038с | 0.047с → 0.075с | шум |
| prev-csv | 1.519с → **0.380с** | 1.656с → 0.480с | **−75%** |

Все ячейки 200×10 (после фиксов; таймаут поднят 30→40с — на 30с medium ловил ложный 502 на p95).

### Находки повторного замера (исправлены по ходу)

1. **unoserver `--user-installation` ждёт путь, не file://-URI** — воркеры не стартовали (502 через acquire-таймаут 10с).
2. **csv требует явного `--input-filter "Text - txt - csv (StarCalc)"`** — иначе UNO открывает его Writer'ом и падает на экспорте в xlsx.
3. **LO-инстанс деградирует после неудачных экспортов** («illegal object given» навсегда) — добавлено самолечение пула: recycle воркера после 2 ошибок подряд + профилактический после 100 джобов.
4. **`--host-location remote` гонит контент base64 через XML-RPC** — заменён на `local` (общая ФС контейнера).
5. ***Medium через UNO на ~25% медленнее прямого soffice** — добавлен гибрид: файлы > `WARM_MAX_BYTES` (2МБ) идут холодным путём (быстрее сами по себе и не занимают тёплых воркеров; контрольный прогон гибрида: 21.6–24.3с — паритет с baseline в пределах шума VM, доминирует сама конвертация).

### Гейт CDN (этап 2) — решение

**Не вычислим и не пройден из доступных точек.** Из FI-точки сетевые стадии пренебрежимы (TLS ~0.02с, health ~0.03с при small-конвертации 0.5с) — сеть ≪ 30%. Для честного гейта нужна точка SEA/US (провижининг VPS — действие владельца: аккаунт/оплата). **Этап 2 (CDN) отложен** до появления дальней точки; серверная часть (тайминг-логи с per-stage разложением) к подсчёту гейта готова.
