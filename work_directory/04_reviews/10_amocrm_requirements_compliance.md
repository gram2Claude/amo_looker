## Контекст

Требования amoCRM написаны в первую очередь для **публичных** интеграций (маркетплейс с модерацией).

разделены: «что важно уже сейчас» и «что станет блокером при выходе в маркетплейс».

## ⚠️ Нарушения и риски

### Актуально уже сейчас

**Н1. `manifest.json`: отсутствует обязательный блок `settings`** (п. 3.1.2: обязательные поля —
`widget`, `locations`, `settings`). Это ещё и **функциональный баг**: `office.js` и `legacy.js`
читают `settings.converter_token` из `get_settings()`, но поле нигде не объявлено → при штатной
установке zip токен в виджет не попадёт (сейчас работает только через букмарклет-путь).
**Фикс:** объявить `settings.converter_token` (text) в manifest.

**Н2. п. 3.2: внешние зависимости через `document.createElement('script')` — прямо запрещено**
(`src/vendorLoader.js:48`). Требование: все зависимости через RequireJS `define`. Усугубляется
занулением `window.define` — манипуляция глобальным окружением кабинета (модерация не пропустит).
Смягчение: скрипт грузится из собственного пакета, не извне; зануление define сериализовано
и с таймаутом. Но формально — нарушение.
**Фикс, закрывающий сразу Н2+Н3:** убрать vendorLoader вообще — поставить `markdown-it` из npm
и заинлайнить в бандл esbuild'ом (`import markdownit from 'markdown-it'`). Цена: бандл вырастет
на ~100 КБ (зип сейчас 60 КБ) — приемлемо.

**Н3. п. 3.1: минифицированный файл в архиве** — `vendor/markdown-it.min.js` («избегать
минимизированные и обфусцированные файлы»). Закрывается тем же фиксом, что Н2.

### Блокеры только для маркетплейса (если когда-нибудь публиковаться)

**М1. п. 2.2 «Изменение интерфейса» — главный архитектурный риск.** Врезка глазиков в ленту
примечаний через MutationObserver + capture-listener на `document` — модификация системного
интерфейса вне разрешённого списка областей. `locations: lcard-1/ccard-1/comcard-1` дают право
на зону виджета в карточке, но не на DOM ленты. Для приватного виджета — ок; на модерации —
высокая вероятность отказа, и это сама суть продукта. Перед подачей — запрос в техподдержку
разработчиков amo о допустимости.

**М2. п. 1.5 / 4.4 — данные и приватность.** Файлы клиентов уходят на сторонний конвертер
(nexus-oko.naithon.one) и в Microsoft Office viewer; shared-токен светится в клиенте.
Для маркетплейса обязательны: ссылка на политику конфиденциальности с реквизитами, раскрытие
в описании (что и куда уходит), и пересмотр схемы токена (общий токен на всех клиентов нельзя).

**М3. п. 4.x — оформление.** Понадобятся: тур-изображения 1188×616, описание «до/после установки»,
инструкция, указание стоимости.

**М4 (minor). п. 2.2** — модалка-оверлей `z-index: 100000` перекрывает весь интерфейс. Открывается
только по действию пользователя и закрывается — риск низкий, но модерация может придраться.

## Вердикт

**Как приватный виджет проект соответствует техническим требованиям amoCRM по существу** — код
чистый по всем формальным пунктам (логи, eval, синхронные запросы, кодировки, стили, i18n,
структура архива, логотипы). Реальных несоответствий три, все локальные: отсутствующий блок
`settings` в manifest (Н1 — заодно прод-баг доставки токена), загрузка vendor через script-тег
(Н2) и минифицированный vendor-файл (Н3). Н2+Н3 закрываются одним ходом — инлайн markdown-it
в бандл, после чего vendorLoader выпиливается целиком.

**Для маркетплейса** проект сейчас не проходной: ключевой риск — сам механизм врезки в ленту
(М1, требует диалога с amoCRM), плюс приватность/токены (М2) и оформление (М3). Но публикация
в маркетплейс и не является целью проекта (прод = приватная установка в кабинет toolkeeper).
